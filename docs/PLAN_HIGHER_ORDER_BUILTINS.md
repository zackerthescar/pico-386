# plan: higher-order PICO-8 builtins via Lua prelude (B) + TFORCALL lua-func support (C)

## context / why

pico-386 is a PICO-8 runtime for DOS/4GW i386: rust no_std compiler (PICO-8 lua → custom
register bytecode, spec in `BYTECODE.md`) + C/nasm VM (`src/p386_dispatch.asm` is the
threaded dispatch core). builtins are CFUNCs registered into fixed global slots
(`include/builtins.h` mirrored by `rust/core_crate/src/bytecode.rs` — these two files are
the wire protocol and MUST stay in sync).

problem: higher-order builtins (`all`, `foreach`) can't be written in C because a CFUNC
cannot re-enter the bytecode interpreter (no reentrant vm_run, and we deliberately don't
want one — it would put C frames inside Lua call chains and block future coroutines).

chosen approach:
- **(B)** implement `all`, `foreach` (and `ipairs`, currently broken — it's registered as
  an alias of `pairs`' CFUNC which is wrong) **in PICO-8 Lua**, as a prelude that the rust
  compiler prepends to every cart source before compiling. the prelude's `function all(t)`
  etc. compile to SETGLOBAL into the reserved builtin slots, overwriting nothing useful
  (those slots will no longer have CFUNCs registered). calling the user callback is then a
  plain CALL instruction — the VM already handles that.
- **(C)** teach `op_tforcall` in `src/p386_dispatch.asm` to accept a TAG_FUNC iterator by
  pushing a normal CallFrame whose `return_reg = A+3`, so `for v in all(t)` works. today
  TFORCALL hard-rejects anything but CFUNC/NIL iterator + TAB state (`err_type_iter`),
  which makes closure-based iterators impossible.

## step 1 — asm: TFORCALL with TAG_FUNC iterator (do this first; independently testable)

current `op_tforcall` decode note: after `mov eax,[esi]; shr eax,16`, `al` is the **B**
byte and that's what holds nvars (the compiler emits `abc(TFORCALL, base, nvars, 0)` —
B carries nvars, despite BYTECODE.md's table saying "A, C". fix the doc while you're there).

generic-for register layout: R[A]=iterator, R[A+1]=state, R[A+2]=control,
R[A+3..A+2+nvars]=loop vars. TFORCALL must call `R[A](R[A+1], R[A+2])` and land results
at R[A+3..]. observe: this is EXACTLY `op_call`'s `.lua_func` shape with function reg = A,
nargs = 2 (args already sit contiguously at R[A+1], R[A+2]), want_rets = nvars — except
returns must go to `A+3` instead of `A`.

implementation: in `op_tforcall`, add a branch: if `R[A].tag == TAG_FUNC`, jump into a
shared "push lua frame" path. cleanest is to factor `op_call`'s `.lua_func` tail into a
subroutine/macro parameterized by (func_reg, nargs, want_rets, return_reg), or set up the
same scratch state op_call uses (`scratch_a` slots: caller A / nargs / want_rets / first-arg
ptr) and jump to a common label — but note the common code derives `return_reg` and the
first-arg pointer from caller A; you need return_reg = A+3 while first-arg ptr = &R[A+1].
so either parameterize those separately or duplicate ~60 lines with the two values changed.
duplication is acceptable if factoring gets hairy — correctness > elegance here, but leave
a comment linking the two copies. `return_ip` = the instruction after TFORCALL (esi already
points there), which is the TFORLOOP — so when the callee RETURNs, dispatch naturally
continues at TFORLOOP, want_rets copy nil-pads missing values (RETURN already does this),
and TFORLOOP's nil-check on R[A+3] terminates the loop. no changes to TFORLOOP or RETURN.

also relax the state type check: for the TAG_FUNC path, R[A+1] (state) and R[A+2] (control)
may be ANY tag (closures ignore them). keep the existing CFUNC/NIL+TAB fast path (pairs)
exactly as is. anything else still traps err_type_iter.

watch out for: call-depth overflow check (CALL_STACK_DEPTH), value-stack overflow check
(callee n_regs), vararg window save — replicate ALL of what `.lua_func` does (frame fields
FRAME_RETURN_IP/BASE/PROTO/CLOSURE/RETURN_REG/WANT_RETS/SAVED_VARARG_*). set
VM_CURRENT_CLOSURE, VM_CURRENT_PROTO, base = old base + caller n_regs, clear callee regs
to nil, copy the 2 args, set up empty vararg window (iterators are not vararg protos in
practice, but handle the flag the same way op_call does).

test (test/test_vm.c, runs via `nix develop -c ./test.sh vm`): hand-build a two-proto
fixture: proto 1 is an iterator closure that upvalue-captures or (simpler) uses state:
easiest deterministic shape without upvalues — iterator `f(state, ctrl)` that returns
`ctrl+1` while `ctrl+1 <= state`, else nil:
  proto1 bytecode: LT check via ADD/LOADK/JMPF etc., RETURN.
main proto: CLOSURE r0; LOADK r1 (state=3); LOADN r2 (control nil→ but control must start
nil per spec; TFORCALL passes it as arg — iterator must handle nil ctrl... to keep the
fixture simple, have the compiler-style setup: MOVE-style init with control = 0 constant
instead of nil is fine since this is a hand fixture and the ASM doesn't inspect control).
then TFORCALL A=0 B=1, TFORLOOP, accumulate loop var into a global, RETURN. assert the
sum. also add: TFORCALL func-path with iterator that returns nothing (want_rets pads nil →
loop exits immediately); TFORCALL on non-func non-cfunc value still traps; call-depth
overflow inside TFORCALL traps err_bounds not crash. use existing fixtures around line
145-320 of test_vm.c as templates for multi-proto containers.

## step 2 — slot plumbing: ALL / FOREACH slots

- `include/builtins.h`: add `P386_BUILTIN_ALL`, `P386_BUILTIN_FOREACH` before
  `P386_BUILTIN_COUNT`. `P386_GLOBAL_INIT` etc. shift automatically (they're
  `= P386_BUILTIN_COUNT` and sequential).
- `rust/core_crate/src/bytecode.rs`: mirror EXACTLY (these are hardcoded `u8` consts —
  ALL=53, FOREACH=54, COUNT=55, GLOBAL_INIT=55, UPDATE=56, UPDATE60=57, DRAW=58,
  USER_GLOBAL_BASE=59). double-check every downstream const.
- `rust/core_crate/src/compiler.rs` `global_slot()`: add `b"all" => P386_BUILTIN_ALL`,
  `b"foreach" => P386_BUILTIN_FOREACH`.
- `src/p386_builtins.c`: do NOT register CFUNCs for ALL/FOREACH — but note
  `p386_builtin_defs[P386_BUILTIN_COUNT]` is a fixed-size array indexed by the loop, and
  `p386_register_builtins` iterates 0..COUNT reading `.slot`. either add entries with a
  NULL func (and skip NULL in the register loop — also make the CALL asm's existing
  `test ebx,ebx / jz err_type_func` guard cover a nil-slot call giving a clean trap), or
  shrink the array. prefer: add entries `{ P386_BUILTIN_ALL, "all", 0 }` and skip-if-NULL
  in the loop, keeping table indices aligned with slots.
- also fix `ipairs`: currently registered as `p386_builtin_pairs` (wrong semantics).
  after the prelude exists, implement `ipairs` in the prelude too (closure-based array
  iterator) and drop its CFUNC registration the same way. keep `pairs` as the CFUNC fast
  path (op_tforcall CFUNC branch depends on it).
- grep test/ for BUILTIN_COUNT / GLOBAL_INIT assumptions (test_vm.c has tests like
  `vm_init_registers_cfunc_builtins` that will need updating for NULL-func slots).

## step 3 — rust: the Lua prelude

- new file `rust/core_crate/src/prelude.lua`, embedded via `include_str!`. contents
  (PICO-8 dialect; verify each construct compiles with p8c BEFORE integrating — the
  compiler is homegrown and may lack sugar):

```lua
function all(t)
 if (t==nil) return function() end
 local i=1
 local prev=nil
 return function()
  if t[i]==prev then i+=1 end
  while t[i]==nil and i<=#t do i+=1 end
  prev=t[i]
  return t[i]
 end
end
function foreach(t,f)
 for v in all(t) do f(v) end
end
function ipairs(t)
 local i=0
 return function()
  i+=1
  if (t[i]~=nil) return i,t[i]
 end
end
```

  notes: `all` must match PICO-8 semantics closely enough that `del(t,v)` inside a
  `foreach` doesn't skip elements (that's what the `prev` compare handles). if the
  compiler lacks `if (x) return y` shorthand or `+=`, rewrite in plain lua — check
  the grammar in `rust/core_crate/src/lib.rs` first. if closures returning multiple
  values from `ipairs`' inner function hit compiler bugs, that's a compiler bug to fix
  or work around (single-var `for v in all(t)` is the critical path; ipairs is nice-to-have
  — drop it from scope if it fights back, but then leave its broken CFUNC registration
  removed and slot nil rather than silently-wrong).

- injection: in `pico386_core::compile()` AND `compile_source()`, prepend
  `concat(PRELUDE, "\n", user_src)` before parsing. CAUTION: the grammar has a
  `header_comment` rule for cart headers — if header comments are only valid at position
  0, prepending breaks carts that start with `-- title` comments. verify with a cart
  source starting with comment lines; if broken, either make the injection insert the
  prelude AFTER leading comment lines, or relax the grammar. test both paths.
- alternative if source-prepend proves fragile: compile the prelude as its own chunk and
  splice protos + prepend its main-chunk bytecode. do NOT go there unless prepend fails —
  it's much more invasive.
- perf note: prelude adds a few protos + constants per cart; negligible.

## invariants / footguns

- builtins.h ↔ bytecode.rs slot values are wire protocol. any drift = silent corruption.
- asm: esi is live IP, ebp is frame base, edi is VMState. `scratch_a`/`dest_tmp` are
  memory scratch — check what the code path you branch from expects to survive.
- everything is 16.16 fixed point: integer n is `n<<16` in globals/constants.
- watcom C is C89-ish: declarations at block top, no `//` comments in .c files
  (match existing style).
- `make` runs outside nix will fail; ALWAYS `nix develop -c`.
- do NOT "fix" p386_vm_call_global's state resets or add reentrant vm_run — out of scope,
  coroutines depend on this staying non-reentrant.
- if qemu tests hang, bump TIMEOUT env; serial log lands in test_serial.log.
